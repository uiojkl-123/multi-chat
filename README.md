# 멀티채팅 서버 (Multi-Chat Server in Rust)

500명 이상이 동시에 접속하는 단체 채팅방을 지원하는 실시간 채팅 서버를 **Rust**로 구현하는 프로젝트.

> 본 문서는 **구현 및 성능 테스트 계획서**다. 실제 측정 결과와 개선 과정은 구현 완료 후 보고서에 작성한다.

---

## 1. 프로젝트 목표

- 단일 채팅방(단톡방)에 500명 이상의 클라이언트가 동시에 접속 가능한 채팅 서버
- 터미널 기반(text mode) CLI 클라이언트
- 메시지 송수신 정합성 검증이 가능한 구조
- 부하 조건 하에서 성능을 측정하고 개선할 수 있는 테스트 환경

---

## 2. 기술 스택

| 구분 | 사용 기술 |
|------|-----------|
| 언어 | Rust (edition 2021) |
| 비동기 런타임 | `tokio` |
| 네트워크 | TCP + 4바이트 길이 접두어(Length-Prefixed) 프레이밍 |
| 직렬화 | `serde` + `serde_json` |
| 동시성 제어 | `tokio::sync::broadcast` |
| 로깅 | `tracing`, `tracing-subscriber` |
| 해시 | `blake3` (메시지 무결성 검증용) |

### WebSocket 대신 Raw TCP를 쓰는 이유

과제 요구사항이 텍스트 모드 클라이언트이고 브라우저 지원이 필수가 아니므로, HTTP 업그레이드 핸드셰이크와 프레임 마스킹 오버헤드가 없는 raw TCP 기반으로 간다.

프레이밍은 4바이트 빅엔디언 길이 접두어만 붙이는 단순한 방식.

## 3. 구현 계획

### 3-1. 전체 구조

```txt
┌──────────────┐           ┌───────────────────────────┐           ┌──────────────┐
│  Client #1   │◄─────────►│                           │◄─────────►│  Client #N   │
└──────────────┘           │          Server           │           └──────────────┘
                           │                           │
┌──────────────┐           │   ┌─────────────────────┐ │           ┌──────────────┐
│  Client #2   │◄─────────►│   │ broadcast::channel  │ │◄─────────►│ Load Tester  │
└──────────────┘           │   │  (fan-out buffer)   │ │           └──────────────┘
                           │   └─────────────────────┘ │
                           └───────────────────────────┘
```

### 3-2. 서버 동작 방식

1. `TcpListener`가 Accept 루프에서 커넥션을 받으면, 커넥션별로 `tokio::spawn`하여 task 할당
2. 각 커넥션은 두 개의 sub-task로 나눔
   - **Reader task**: 소켓 → 서버 브로드캐스트 채널
   - **Writer task**: 서버 브로드캐스트 채널 → 소켓
3. 브로드캐스트는 `tokio::sync::broadcast` 채널 하나로 팬아웃
4. broadcast 버퍼가 가득 차서 `Lagged` 발생 시, 해당 클라이언트는 slow consumer로 보고 연결 종료

### 3-3. 메시지 프로토콜

프레임:

```txt
[4바이트 length (BE)] [JSON payload]
```

```json
{
  "type": "Chat",
  "msg_id": "c001-0042",
  "from": "client-001",
  "ts": 1737270000123,
  "hash": "a3f1...9c",
  "body": "안녕하세요"
}
```

| 필드 | 설명 |
|------|------|
| `type` | `Join` / `Leave` / `Chat` / `Ack` / `Sys` |
| `msg_id` | `{clientId}-{seq}` 형식. 클라이언트가 생성 |
| `from` | 송신자 ID |
| `ts` | 송신 시각 (Unix ms) |
| `hash` | `body`의 BLAKE3 해시 (무결성 검증용) |
| `body` | 실제 메시지 본문 |

### 3-4. 디렉터리 구조 (Cargo Workspace)

```txt
multichat/
├── Cargo.toml
├── crates/
│   ├── protocol/   # 공통 메시지 정의 (서버/클라이언트 공유)
│   ├── server/     # 채팅 서버
│   ├── client/     # CLI 클라이언트
│   └── loadtest/   # 부하 생성기 + 검증 도구
└── README.md
```

`protocol` 크레이트를 공유해 서버/클라이언트 간 타입 정의를 일치시킨다.

## 4. 메시지 송수신 검증 계획

성능만큼이나 "제대로 보내고 받았는지" 검증이 중요하다. 세 단계로 검증한다.

### 4-1. 메시지 무결성 (Hash 검증)

송신 시 `body`의 BLAKE3 해시를 `hash` 필드에 실어 보낸다.

수신 측에서 다시 해싱해 비교하여 손상 여부를 확인한다.

### 4-2. 메시지 누락 (Sequence 검증)

`msg_id = "{clientId}-{seq}"` 형식을 활용한다.

각 수신자는 송신자별 시퀀스 번호의 연속성을 검사한다.

500명이 각자 N개 메시지를 보낸다면, 각 클라이언트는 나 자신을 제외한 `499 × N`개를 빠짐없이 받아야 한다.

### 4-3. End-to-End 일관성

부하 테스트 종료 후, 각 클라이언트가 수신한 메시지 집합을 덤프해 비교한다.

- 모든 클라이언트가 동일한 **집합**을 받았는가
- 총 송신량과 총 수신량이 `송신자 수 × (수신자 수 - 1)` 관계를 만족하는가

> 순서는 각 클라이언트의 소켓 쓰기 타이밍에 따라 달라질 수 있으므로, **집합 단위**로 검증한다.

## 5. 성능 테스트 계획

### 5-1. 측정 지표

| 지표 | 정의 | 측정 방법 |
|------|------|-----------|
| **Latency (P50/P95/P99)** | 송신 시각 → 타 클라이언트 수신 시각 | `msg.ts`와 수신 시각의 차이 |
| **Throughput** | 초당 서버가 처리한 메시지 수 | `AtomicU64` 카운터를 1초 주기로 샘플링 |
| **Loss Rate** | (송신 총량 - 수신 총량) / 송신 총량 | 시퀀스 검증 결과 기반 |
| **CPU / Memory** | 서버 프로세스 리소스 사용량 | `htop`, `/proc/self/status` |

### 5-2. 테스트 시나리오

| 시나리오 | 클라이언트 수 | 메시지 레이트 | 지속 시간 | 목적 |
|----------|---------------|---------------|-----------|------|
| S1 (기준) | 100 | 1 msg/s | 60s | 기본 동작 확인 |
| S2 (요구사항) | 500 | 1 msg/s | 60s | 과제 요구사항 충족 여부 |
| S3 (스트레스) | 500 | 10 msg/s | 30s | 고부하 시 동작 확인 |
| S4 (번인) | 500 | 2 msg/s | 10min | 장시간 안정성 |

### 5-3. 개선 전략 (예상 포인트)

실제 프로파일링 결과에 따라 조정하겠지만, 다음 지점을 중점적으로 살펴볼 계획이다.

1. **락 경합**: 클라이언트 관리 구조를 `Mutex`로 시작했을 때 병목이 될 수 있음 → `broadcast` 채널 기반으로 대체
2. **Slow consumer**: 느린 클라이언트 하나가 전체 tail latency를 끌어올리는 현상 → `Lagged` 발생 시 연결 종료 정책 적용
3. **직렬화 중복**: 동일 메시지를 N번 직렬화하는 비효율 → 한 번만 직렬화해서 `Bytes`로 refcount 공유
4. **버퍼 크기 튜닝**: broadcast 채널 용량, TCP `SO_SNDBUF`/`SO_RCVBUF` 조정

측정 → 병목 식별(`perf`, `flamegraph`) → 개선 → 재측정 사이클로 진행한다.

## 6. 실행 방법

### 6-1. 사전 준비

실행 전에 Rust stable toolchain이 설치되어 있어야 한다.

```bash
rustc --version
cargo --version
```

또한 모든 명령어는 프로젝트 루트 디렉터리에서 실행한다.

### 6-2. 서버 실행

먼저 채팅 서버를 실행한다.

```bash
cargo run -p server --release -- --addr 0.0.0.0:9000
```

기본 포트는 `9000`이며, 이미 사용 중인 포트라면 `--addr` 값을 변경해서 실행할 수 있다.

### 6-3. 클라이언트 실행

서버가 실행 중인 상태에서 별도 터미널을 열고 클라이언트를 실행한다.

```bash
cargo run -p client --release -- --addr 127.0.0.1:9000 --name martin
```

`--name` 값은 채팅에서 사용할 클라이언트 이름이다.

### 6-4. 부하 테스트 실행

서버가 실행 중인 상태에서 부하 테스트 도구를 실행한다.

```bash
cargo run -p loadtest --release -- \
    --addr 127.0.0.1:9000 \
    --clients 500 \
    --rate 1 \
    --duration 60
```

위 예시는 500명의 클라이언트가 초당 1개 메시지를 60초 동안 전송하는 테스트다.

### 6-5. 결과 보고서 자동 생성 (개선 전/후 비교)

`bench/` 디렉터리에 §5-2의 시나리오를 자동으로 돌리고, 개선 전/후 결과를 합쳐 마크다운 보고서로 만들어주는 PowerShell 스크립트가 들어 있다.

먼저 베이스라인을 측정한다 (서버는 스크립트가 직접 띄웠다 내림).

```powershell
.\bench\run-scenarios.ps1 -Label baseline
```

이후 서버/프로토콜을 개선하고 다시 측정한다.

```powershell
.\bench\run-scenarios.ps1 -Label improved
```

각 실행은 `bench\results\<라벨>-<시나리오ID>.json` 형식으로 결과를 남긴다. 두 라벨이 다 모이면 비교 보고서를 만든다.

```powershell
.\bench\build-report.ps1 -Baseline baseline -Improved improved
# -> bench\results\report.md
```

보고서는 시나리오별 요약 표(P50/P95/P99/Loss/오류의 baseline → improved Δ)와 각 시나리오의 상세 표를 포함한다. 그대로 결과 보고서에 붙여 넣을 수 있다.

옵션:

- `-Scenarios S1,S2` : 일부 시나리오만 돌리기 (기본은 S1~S4 전부)
- `-KeepServer`     : 이미 띄워둔 서버를 사용하고 스크립트가 서버를 건드리지 않게 함
- `-Addr 127.0.0.1:9000` : 서버 주소 (기본 동일)

`loadtest` 바이너리 자체에도 `--output text|json|md` 와 `--label <name>` 옵션이 추가되어 있어, 스크립트 없이 단발 측정을 보고서용으로 뽑고 싶다면 직접 호출해도 된다.

```powershell
cargo run -p loadtest --release -- `
    --addr 127.0.0.1:9000 --clients 500 --rate 1 --duration 60 `
    --output md --label improved-S2
```
